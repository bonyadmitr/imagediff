# ImageDiff — инженерный журнал

Документ для того, чтобы **не делать двойную работу**: что уже есть, как оно работает, какие гипотезы проверяли и что из них вышло — **включая неудачные попытки и тупики**. Всё ниже — по фактам из разработки (замеры, поведение на реальных примерах), без догадок. Где вывод основан на счётчиках без поштучной визуальной сверки — это отдельно отмечено.

Дата составления: 2026-06-30.

Связанные документы: [дизайн-спека](superpowers/specs/2026-06-29-image-diff-design.md), [план](superpowers/plans/2026-06-29-image-diff.md), [README](../README.md) (пользовательский), скил деплоя `.claude/skills/deploy-github-pages/`.

## 3. Как работает конвейер (v1.1)

Вход: одно RGBA-изображение (как `ImageData`). Выход: `{status, orientation, offset:{dx,dy}, panelA, panelB, diffs:[{x,y,w,h,score}]}`. Координаты отличий — в системе картинки A; чтобы нарисовать на картинке B, прибавить `offset`.

1. **Поиск смещения и ориентации** (`align.js`, на полутоне).
   Две картинки — одна сцена, сдвинутая на постоянный вектор. Сравниваем изображение с собственной сдвинутой копией при разных сдвигах, берём минимум среднего модуля разности (MAD). Минимум даёт **и ориентацию** (по направлению вектора), **и расстояние**. Делается в два прохода:
   - **грубо** на сильно уменьшенной копии (`alignMaxDim=160`);
   - **уточнение** на умеренно уменьшенной копии (`alignRefineDim=700`), а НЕ на полном разрешении (см. раздел 5.3 — это была ключевая оптимизация скорости).
   Если лучшее совпадение хуже `alignMaxScore=70` — считаем, что две картинки не найдены (`panels_not_found`).

2. **Рабочий масштаб** (`maxWorkingDim=1100`). Картинку уменьшают так, чтобы бóльшая сторона не превышала это значение — стадия различий идёт быстрее. Рамки потом масштабируются обратно в координаты оригинала.

3. **Признаки**: для рабочего масштаба берём каналы R, G, B (каждый размывается `boxBlur`, `blurRadius=1`) и карту контуров `gradientMag` (|gx|+|gy| по полутону).

4. **Толерантная к сдвигу карта различий** (`diff.js`, `tolerantDiffField`).
   Для каждого пикселя картинки A берём по окрестности радиуса `matchRadius=2` вокруг соответствующей точки в B **минимальное** расстояние, где расстояние = **взвешенный максимум** модулей разностей по каналам: `max(|ΔR|, |ΔG|, |ΔB|, edgeWeight·|Δedge|)`, `edgeWeight=0.6`. Минимум по окрестности гасит «дрожание» перерисованных/пересжатых контуров; максимум по каналам ловит и перекраски (цвет), и изменения формы (контуры).

5. **Зона панелей по доле совпавшего контура** (`findPanelZone`). Содержательный пиксель = есть контур (`edge ≥ edgeContentThr=18`). Для каждой строки и столбца считаем **долю совпавших среди содержательных**. В самих картинках контур в основном совпадает (доля высокая ≈0.9), в тексте-заголовке/мини-ответе контур НЕ совпадает (доля низкая) → отсекается. Берём самую длинную непрерывную полосу строк и столбцов, где:
   - перекрытие достаточно (`зона` реально внутри overlap, `zoneMinValidFrac=0.3`), И
   - доля совпавшего ≥ `zoneRatioThr=0.55` (строки с малым числом контуров `zoneMinContentFrac=0.04` считаем фоном панели и включаем).
   Если контуров почти нет (гладкая картинка) — **фолбэк**: крупнейшая просто совпадающая область.

6. **Маска отличий** внутри зоны: пиксели, где различие ≥ `threshold=50`. Чистка: `open` (убрать одиночный шум) + `dilate` (слегка нарастить).

7. **Прямоугольники + ранжирование**: связные компоненты (фильтр по `minBlobFrac=0.0005`), слияние близких (`mergeGapFrac=0.012`). Каждая рамка получает **силу** = средняя величина различия × √(площадь). Сортировка по убыванию силы; `maxResults` оставляет топ-N.

### Все параметры (`DEFAULT_OPTIONS`)
Полная таблица с диапазонами и пояснениями — в [README](../README.md#все-параметры-default_options). Текущие значения: `threshold 50, matchThreshold 22, matchRadius 2, edgeWeight 0.6, blurRadius 1, maxWorkingDim 1100, minBlobFrac 0.0005, mergeGapFrac 0.012, maxResults null, maxDiffs 60, edgeContentThr 18, zoneRatioThr 0.55, zoneMinContentFrac 0.04, zoneMinValidFrac 0.3, alignMaxDim 160, alignRefineDim 700, alignStep 2, refineStep 2, minSeparationFrac 0.15, minOverlapFrac 0.2, maxSkewFrac 0.04, alignMaxScore 70`.

## 4. История гипотез и результатов

### 4.1. Платформа: PWA против нативки — выбрана PWA
Для личного инструмента PWA выигрывает: вес на порядки меньше (десятки КБ против десятков МБ), раздача одной ссылкой, один код на iOS/Mac/везде, быстрые итерации. Нативка дала бы интеграцию с Фото/камерой и максимум производительности — в этом сценарии не нужно. Дизайн-док при этом написан универсальным, ядро переносимо.

### 4.2. «Резать картинку пополам» — ОТВЕРГНУТО
Первоначальное допущение: кадр = две равные половины. **Не подтвердилось** на реальных примерах пользователя: в кадре бывает лишнее — баннеры («Найди 10 отличий»), знак «?», и даже **мини-ответ** (уменьшенная копия с уже обведёнными отличиями). Перешли на поиск вектора смещения через само-сопоставление: лишнее не повторяется/другого масштаба и не даёт сильного совпадения.

### 4.3. Синтетический фон в тестах — дважды ломал тесты (отрицательный опыт)
- **Периодический фон.** Первый генератор: градиент по X + клетка с периодом 16 по обеим осям. Вертикальный сдвиг на dy=64 (кратно 16) идеально совмещал клетку, а градиент по X от вертикального сдвига не менялся → **ложное «идеальное» совпадение**, движок путал ориентацию. Урок: фон должен быть **непериодическим**. Заменили на сумму несоизмеримых синусов.
- **Квазипериодичность синусов.** Сумма синусов всё равно квазипериодична: при панели высотой 1000 px вертикальный сдвиг ~628 совмещал все синус-компоненты → в одном **node-замере** (не в юнит-тестах) движок выбрал ложную вертикаль. В юнит-тестах (панель 90 px) проблемы нет. Урок: синтетика — управляемая фикстура, но её периодичность надо держать в голове; реальные картинки не периодичны.

### 4.4. Производительность: было ~5 c на крупной картинке → стало ~0.1–0.2 c
- Первый прогон nukadeti (1500×2052): **5044 мс**. Профилирование: `findOffset` = ~3.9 c.
- Изолировали причину **замерами** (а не догадками): `refineStep:6` → 523 мс (значит, дорого именно уточнение); `alignMaxDim:80` → 14626 мс (хуже! т.к. окно уточнения растёт как фактор²). Вывод: **уточнение смещения шло на полном разрешении**, а это сотни замеров MAD по миллионам пикселей.
- Фикс: уточнение на ограниченном масштабе (`alignRefineDim=700`). Замер в Node на 10 реальных примерах: **47–235 мс**.
- Побочно: коэффициент рабочего масштаба считался через `round` → для 1500 px давал 1 (без уменьшения). Заменили на `ceil` → 1500 px уменьшается до 750 (вчетверо меньше пикселей в стадии различий).

### 4.5. Браузерный preview-рендерер — таймауты оказались артефактом инструмента, НЕ кодом
Симптом: `preview_eval`, которые декодируют JPEG и гоняют движок, висели >30 c. Проверка по границам:
- `import` модуля в браузере — **9 мс** (код грузится, свежий).
- Тот же движок в **Node** — **54 мс** на 1460×1000.
Вывод: JS из «отвалившихся» по клиентскому таймауту вызовов **продолжает крутиться** и забивает поток headless-рендерера; новые вызовы стоят в очереди и тоже таймаутятся. Это особенность инструмента, не приложения (первый e2e и первые прогоны отрабатывали и рисовали). **Решение:** для замеров/проверки качества — детерминированный harness в Node (см. раздел 7), браузер — только для финального визуального контроля одним свежим вызовом.

### 4.6. Поиск зоны панелей — 4 неудачных подхода до рабочего (главный тупик сессии)
Задача: исключить баннер-текст и мини-ответ, оставив сами картинки. Перепробовано по порядку:
1. **Крупнейшая просто совпадающая область** (`matchMask`). Провал на штриховых картинках по белому фону: белое «совпадает» везде → компонента и её bbox охватывают весь кадр вместе с баннером.
2. **Крупнейшая область совпавшего КОНТЕНТА** (контур И совпал) + дилатация. Провал: баннер даёт ~7% случайно совпавшего контента (замерено: 321 из 4800 пикселей), дилатация-«мостик» притягивает его к панели → bbox снова охватывает баннер.
3. **Плотность совпавшего контента по строкам/столбцам.** Провал на гладком синтетическом фоне: контуры неравномерны → полоса схлопывается в узкий диапазон, теряются отличия.
4. **Доля совпавшего среди содержательного, без учёта перекрытия.** Провал: при разреженных контурах доля шумит; области ВНЕ перекрытия имели «долю=1» и выигрывали как «включённые» → выбиралась не та панель.
5. **РАБОЧИЙ вариант:** доля совпавшего среди содержательного + порог минимального контента (разреженные строки = фон, включаем) + **порог перекрытия** (вне overlap не включаем). Прошли все 20 тестов; на реальных примерах баннеры и мини-ответ перестали обводиться. Урок: связная компонента чувствительна к «мостикам»; устойчивее — статистика по строкам/столбцам с явной отбраковкой зон вне перекрытия.

### 4.7. Цвет и контуры — закрыли недобор на цветных
В v1 сравнение шло по **яркости** (grayscale). Провал на цветных: перекраски с похожей яркостью (красный↔зелёный) в полутоне почти не видны (sample 5 — нашлось 1 отличие). Фикс: сравнение по каналам R,G,B (максимум модулей) + канал контуров. После — sample 5: **5 отличий** (проверено визуально — рамки на реальных различиях).

### 4.8. nukadeti — разбор «гигантской рамки» (диагностика по тепловой карте)
Симптом: одна рамка накрывала почти всю панель. Проверки:
- Смещение **верное** (скан по dy дал глобальный минимум на 1024; не ошибка выравнивания).
- Толерантность ±3 не помогла — значит, не субпиксельный промах.
- Тепловая карта различий показала: **83%** пикселей совпадают, сильно отличаются **14%**, и они **сосредоточены** на реальных отличиях (флаг красный↔белый, переехавший космонавт, планета, «тарелка», луноход). То есть детектирование верное, но крупные отличия (переехавший объект) сливались в один большой бокс на стадии сборки. После v1.1 (толерантность + зона) — лучше, но nukadeti остаётся самым трудным (см. раздел 6).

## 6. Известные ограничения и что работает НЕ идеально

- **Недобор на штриховых картинках с тонкими отличиями.** Sample 6: задание «найди 9», движок уверенно даёт 3 (без ложных). Тонкие правки по линиям при пороге `threshold=50` и толерантности `matchRadius=2` подавляются. Частично добирается слайдером чувствительности.
- **nukadeti — самый трудный** (3 отличия). Крупные «переезды» объектов и перекраска большой области.
- **Баннеры с большим количеством случайно совпадающего контента** теоретически могут частично попасть в зону (механизм статистический, не идеальный).
- **Только чистые цифровые картинки.** Фото распечаток (перспектива, блики, перекос) — вне скоупа; текущее выравнивание только по сдвигу (без поворота/масштаба).
- **Полуручной режим (обведение двух панелей) — не реализован в UI.** В ядре уже есть `manualRegions` (контракт готов), кнопка-заглушка в интерфейсе присутствует.
- **Сравнение по `max` каналов RGB**, а не по перцептивному ΔE (Lab) — простое и работает, но не самое точное по цвету.
- **Service worker кэширует версию.** После редеплоя вернувшиеся пользователи могут видеть старую версию, пока SW не обновится; имя кэша версионируется (`imagediff-vN` в `app/sw.js`) — бампать при критичных изменениях.

## 7. Инструменты разработки и проверки (чтобы воспроизвести)

- **Тесты ядра:** `npm test` (нужен только Node ≥ 18). Тестируем по реальным рискам (ориентация/смещение, попадание рамок, цвет, контуры, ранжирование, краевые случаи), без погони за покрытием.
- **Детерминированная проверка на реальных JPEG без браузера** (надёжнее капризного preview):
  1. `sips -s format bmp samples/N.jpg --out scratch/N.bmp` — JPEG → BMP (BMP легко читать в Node без зависимостей);
  2. `annotate.mjs` — читает BMP, гоняет `findDifferences`, рисует рамки на обеих панелях, пишет BMP, печатает JSON (статус, ориентация, смещение, число рамок, время);
  3. `sips -s format png scratch/N.out.bmp --out scratch/N.out.png` — BMP → PNG, чтобы посмотреть глазами.
  Скрипт `annotate.mjs` лежал в scratchpad сессии (не в репозитории); при необходимости пересоздаётся по этому описанию (читалка/писалка 24-битного BMP + рисование прямоугольников в RGBA).
- **Браузер:** только для финального визуального контроля. Перед прогоном свежего кода — снять регистрацию service worker и очистить `caches`, затем перезагрузить страницу; гонять по одному лёгкому вызову (большие/множественные тяжёлые вызовы забивают поток рендерера).

## 8. Анализ: даст ли OpenCV ощутимый прирост?

Короткий вывод: **для текущего скоупа (чистые цифровые картинки) — прирост скромный**; основная ценность OpenCV — будущий скоуп «фото».

| Стадия | Что добавил бы OpenCV | Прирост сейчас |
|---|---|---|
| Выравнивание | ORB/SIFT+homography, `findTransformECC`, phase correlation | ~0: смещение и так верно на всех 10; это оружие против поворота/перспективы (т.е. для фото). |
| Сигнал различия | Lab/ΔE, SSIM, bilateral | малый: уже сравниваем цвет+контуры с допуском; SSIM/ΔE дали бы единицы процентов. |
| Морфология | те же erode/dilate/open | ~0 по качеству. |

- **Остаточные проблемы** (тонкие штриховые отличия, nukadeti) — про тонкость/семантику различий и компромисс порога, а не про мощность библиотеки. OpenCV считает ту же пиксельную математику.
- **Цена:** OpenCV.js ≈ **8 МБ wasm** (урезанный кастом-билд ~1–3 МБ) — ломает плюс «лёгкий PWA».
- **Где окупится:** фото распечаток (перспектива, блики, перекос) — устойчивое выравнивание это его коронная задача.

**Вместо полного OpenCV для текущих картинок** (тот же эффект, чистый JS, без 8 МБ): SSIM, адаптивный/локальный порог (Otsu/локальное среднее), ΔE в Lab. Грубая оценка: SSIM + адаптивный порог ≈ **+10–20%** пойманных настоящих отличий на штриховых. Это стоит делать раньше OpenCV; OpenCV — строго на фото. (Числа — оценка, не замер; точную проверку можно сделать SSIM-прототипом на тех же 10 примерах.)

## 10. Грабли и уроки (чтобы не наступить снова)

- **zsh не разбивает `$var` по пробелам** (в отличие от bash). Список имён файлов в цикле передавать массивом `names=(...)`, иначе вся строка уходит одним аргументом.
- **Кэш ES-модулей + service worker** в браузере: `?v=N` на entry-модуле НЕ бастит его внутренние относительные `import` — они тянутся из кэша. Для честной проверки свежего кода — снять SW и очистить caches, потом перезагрузить (новый realm).
- **Клиентский таймаут eval ≠ остановка JS.** Зависший тяжёлый вызов продолжает занимать поток рендерера и валит последующие вызовы. Не множить тяжёлые вызовы; проверять в Node.
- **`round` vs `ceil`** при выборе коэффициента уменьшения: `round(1500/1100)=1` (без уменьшения!) — для гарантии «не больше N» нужен `ceil`.
- **Синтетика не должна быть периодичной** — иначе ложные совпадения при сдвиге ломают определение ориентации.
- **Связная компонента чувствительна к «мостикам»** из единичных выбросов; для выделения области устойчивее статистика по строкам/столбцам с явной отбраковкой.
- **Уточнять смещение на полном разрешении дорого** (окно растёт как фактор²); делать на ограниченном масштабе — точности с запасом хватает, т.к. дальше есть допуск `matchRadius`.

### Почему адрес оседает на `…/imagediff/app/`, а не `…/imagediff/`
Само приложение лежит в подпапке `app/` (сохраняем разделение слоёв: переносимое ядро `src/core/` отдельно от адаптера `app/`, который импортирует ядро как `../src/core/…`). В корне репозитория — только `index.html` с `<meta refresh>`, который перебрасывает на `app/`. Поэтому голый адрес `…/imagediff/` **работает** (открывает приложение через редирект), но в адресной строке оседает `…/imagediff/app/`.

Чтобы приложение отдавалось прямо на `…/imagediff/` (как в проектах с `index.html` в корне), нужно перенести файлы приложения в корень и поправить относительные пути (`../src/core/…` → `./src/core/…`, список кэша в `sw.js`, пути в `manifest.webmanifest`), удалив редирект. **Решено оставить как есть — ради простоты и сохранения разделения слоёв** (осознанный выбор, 2026-06-30).
