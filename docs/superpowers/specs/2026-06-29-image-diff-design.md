# Дизайн: ImageDiff — поиск отличий на двух картинках

**Дата:** 2026-06-29
**Статус:** утверждён к реализации
**Тип:** личный инструмент (без публикации в App Store)

---

## 1. Цель и контекст

Приложение принимает **одно изображение**, на котором расположены **две почти одинаковые картинки** (классическая головоломка «найди отличия»), автоматически находит все отличия и **подсвечивает их рамками**.

Картинки внутри кадра могут располагаться:
- **горизонтально** — две панели рядом (левая/правая);
- **вертикально** — две панели одна над другой (верхняя/нижняя).

Кроме самих двух картинок в кадре может присутствовать **лишний контент**: баннеры и надписи («FIND 10 DIFFERENCES», «Найди 10 отличий»), декоративный фон, знак «?», и даже **мини-ответ** — уменьшенная копия сцены с уже обведёнными отличиями. Этот лишний контент нужно игнорировать; мини-ответ особенно важно не принять за одну из основных картинок.

### Сценарий использования
Личный инструмент для себя и друзей. Дистрибуция — простая (ссылка). Главное — чтобы надёжно работало, без затрат на сторы и подписи.

### Что в скоупе v1
- Только **чистые цифровые картинки** (скриншоты, скачанные изображения). Без искажений геометрии.
- **Полностью автоматический** режим.
- Режим вывода — **рамки** вокруг каждого отличия.

### Что отложено на будущее (НЕ в v1)
- Фото распечаток/книг (перспектива, блики, тени, перекос, шум).
- Полуручной режим (пользователь сам обводит две панели). **Но «аварийный люк» под него закладываем в архитектуру и UI сразу.**
- Режимы вывода «заливка» и «по одному».
- Нативные приложения iOS/Mac (дизайн-док пишется универсальным именно ради этого).

## 2. Выбор платформы

**Решение: PWA (чистое клиентское веб-приложение, без сервера).**

Вся обработка идёт локально в браузере. Картинку загружаем через выбор файла / drag-and-drop / вставку, обработка — на Canvas.

**Почему PWA, а не нативка (для этого сценария):**
- **Вес:** на порядок меньше. Для чистых картинок тяжёлые библиотеки (OpenCV) не нужны — хватает Canvas и собственного кода на десятки килобайт.
- **Доставка друзьям:** просто ссылка (или один HTML-файл). Без Xcode, подписи, TestFlight, App Store.
- **Один код** работает на iOS, Mac и где угодно. Можно «установить» на домашний экран айфона и в Dock на Mac.
- **Скорость итераций:** максимальная.
- Производительности Canvas для маленьких картинок более чем достаточно; офлайн обеспечивается service worker.

Нативка дала бы лучшую интеграцию с Фото/камерой и максимум производительности — но в этом сценарии это не нужно.

## 3. Архитектура

Два чётко разделённых слоя. **Граница между ними — главная ценность дизайна:** ядро переносимо на любую платформу без изменений спецификации.

### 3.1. Ядро `diffEngine` — платформонезависимое

Чистые функции, ничего не знающие про браузер, DOM или Swift. Работают только с «сырыми» пикселями.

**Контракт (концептуально):**

```
Вход:
  image: { data: Uint8ClampedArray (RGBA), width, height }
  options: {
    threshold,            // порог чувствительности (см. §4)
    orientation,          // 'auto' | 'horizontal' | 'vertical'
    manualRegions?,       // опционально: две заданные вручную области (аварийный люк)
    minBlobArea,          // минимальная площадь пятна, чтобы считать его отличием
  }

Выход:
  {
    orientation,          // что в итоге определили
    offset: { dx, dy },   // вектор смещения между панелями
    panelA: { x, y, w, h },
    panelB: { x, y, w, h },
    diffs: [ { x, y, w, h } ],   // прямоугольники отличий в координатах исходного изображения
    status,               // 'ok' | 'panels_not_found' | 'no_diffs' | 'too_many_diffs'
  }
```

Ядро **не рисует** ничего и не трогает UI — только считает и возвращает данные.

### 3.2. Платформенный адаптер

Сейчас — **Canvas + DOM (PWA)**. Потом — **CoreGraphics + SwiftUI (нативка)**. Отвечает за:
- загрузку картинки в пиксельный буфер;
- передачу буфера и опций в ядро;
- отрисовку рамок поверх картинки по координатам из ответа ядра;
- сохранение/шаринг результата;
- UI и его состояние.

**Аварийный люк (ручное выделение):** контракт ядра принимает `manualRegions`. Даже если в v1 UI ручного выделения ещё не реализован, ядро уже умеет работать с заданными областями — нужно лишь подключить ввод. Это и есть заложенная страховка под полуручной режим.

## 4. Алгоритм поиска отличий (конвейер ядра)

Ключевая идея: **не резать кадр пополам**, а найти, на какой вектор сдвинута одна копия относительно другой, через **само-сопоставление** изображения с собственной сдвинутой копией.

### Шаг 1. Найти смещение между двумя копиями
Две картинки-головоломки — это одна сцена, сдвинутая на постоянный вектор (вправо или вниз). Сравниваем изображение с собственной сдвинутой копией при разных кандидатах сдвига и берём **самый сильный пик совпадения** (нормализованная корреляция / минимум суммарной разности на уменьшенной серой копии — для скорости).

Этот пик сразу даёт:
- **ориентацию** (горизонтальная или вертикальная) — по направлению вектора;
- **расстояние между панелями** — по величине вектора.

Почему лишний контент не мешает:
- баннеры и надписи **не повторяются** → сильного пика не дают;
- мини-ответ — это **уменьшенная** копия → на исходном масштабе не совпадает с панелями;
- значит, доминирующий пик — именно две полноразмерные панели.

Если `orientation` задан явно (не `auto`), ищем сдвиг только в нужном направлении. Если заданы `manualRegions` — шаг 1 пропускаем, смещение и панели берём из них.

### Шаг 2. Определить зону сравнения
Накладываем изображение на его сдвинутую (на найденный вектор) копию. **Зона сравнения** — область, где они хорошо совпадают. Участки, которые «не совпали вообще» (баннеры, надписи, мини-ответ), в зону не попадают и отбрасываются. Из зоны выводим прямоугольники `panelA` и `panelB`.

### Шаг 3. Карта различий
Внутри зоны сравнения:
1. лёгкое размытие (гасит «лесенку» сглаживания/анти-алиасинг);
2. попиксельная разница панели A и панели B;
3. порог (`threshold`) → бинарная маска изменённых пикселей.

### Шаг 4. Чистка маски
Морфология: убрать мелкий «мусор» (эрозия), склеить близкие пиксели (дилатация). Так разрозненные пиксели одного отличия собираются в одно пятно.

### Шаг 5. Связные области → прямоугольники
Находим связные компоненты маски, отбрасываем слишком мелкие (площадь < `minBlobArea`), сливаем перекрывающиеся/близкие. Получаем итоговый список рамок `diffs`.

### Шаг 6. Возврат результата
Координаты рамок — в системе исходного изображения. Каждое отличие присутствует на обеих панелях (одни координаты внутри панели, со сдвигом на вектор между панелями) — адаптер может нарисовать рамку на обеих.

## 5. UX / поток работы (PWA, один экран)

1. **Загрузка картинки** — выбор файла, drag-and-drop, вставка из буфера (Cmd/Ctrl+V); на телефоне — выбор из Фото. Зона «Загрузить».
2. **Авто-обработка** — сразу после загрузки прогоняем конвейер, показываем картинку с рамками и счётчик «Найдено: N».
3. **Подстройка** (минимум рычагов):
   - **Слайдер чувствительности** (`threshold`) — если нашлось мало или много.
   - **Переключатель ориентации**: авто / горизонт. / вертик. — страховка от неверного автоопределения.
   - **«Выделить вручную»** — кнопка присутствует в UI с первого дня (аварийный люк); функционал ручного обведения двух панелей можно включить позже.
4. **Результат** — сохранить картинку с рамками / поделиться (Web Share API) / «Загрузить другую».

## 6. Обработка ошибок и краевые случаи

| Ситуация | Поведение |
|---|---|
| Не нашли смещение/панели (`panels_not_found`) | «Не удалось распознать две картинки автоматически. Попробуйте выделить их вручную» + подсветка кнопки ручного режима |
| Отличий не найдено (`no_diffs`) | Подсказка снизить порог |
| Слишком много пятен (`too_many_diffs`) | Подсказка поднять порог (вероятно шум или не та картинка) |
| Не картинка / битый файл | Понятное сообщение |
| Очень большое изображение | Масштабируем перед обработкой (ограничение по стороне) |
| Мини-ответ с обведёнными отличиями в кадре | Не попадает в зону сравнения (меньший масштаб → нет пика совпадения) |

## 7. Тестирование

**Принцип: качество, а не покрытие.** Тесты проверяют реальные риски и поведение, без погони за процентами.

**Юнит-тесты ядра** (платформонезависимые, переносятся вместе с алгоритмом) — на синтетических картинках с заранее известными отличиями:
- верно определены **ориентация** и **вектор смещения** (и для горизонтального, и для вертикального расположения);
- найдены именно заданные отличия (рамки попадают в нужные места), нет ложных;
- лишний контент (баннер, надпись, мини-ответ) **не** принят за панель и не даёт ложных отличий.

**Набор реальных примеров** — картинки пользователя (обезьянка в машине — горизонтально, инопланетяне — вертикально, с баннерами) и ещё несколько разных. Визуальная проверка результата.

**Краевые случаи** — без отличий, очень много отличий, не та картинка, лишний контент по краям.

## 9. Доработки v1.1 (сделано после проверки на реальных картинках)

После прогона на реальных головоломках исходный «попиксельный по яркости» подход показал два изъяна: ложные рамки на тексте-заголовках/мини-ответе и недобор на цветных перекрасках. Доработано:
- **Сравнение по цвету и контурам** (RGB + карта границ) вместо только яркости — ловит перекраски с похожей яркостью и изменения формы.
- **Толерантная к сдвигу разница** (минимум по окрестности) — гасит «дрожание» перерисованных контуров.
- **Зона панелей по доле совпавшего контура** по строкам/столбцам — исключает баннеры и мини-ответ (с учётом перекрытия).
- **Ранжирование отличий по силе + топ-N**, удаление лишней рамки нажатием.
- **Уточнение смещения на ограниченном масштабе** — крупные картинки обрабатываются за ~0.1–0.2 c.

## 10. Открытые вопросы на будущее
- Поддержка фото (перспектива/блики) — вероятно потребует выравнивания с поворотом и, возможно, OpenCV.js (~8 МБ wasm).

- Нативные приложения iOS/Mac на базе того же ядра.
